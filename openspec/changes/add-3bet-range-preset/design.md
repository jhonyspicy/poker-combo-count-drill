## Context

プリセットレンジは `src/lib/presetRanges.ts` の `PRESET_RANGES` 配列に定義され、選択（`pickPresetRange`）・グリッド展開（`presetToGridTest`）・コンボ計算（`presetComboCount`）はすべて配列駆動で動く。現在は「オープンレンジ」（216コンボ）と「コールレンジ」（118コンボ）の2種。合計コンボ数は必ずロジックで計算し、定義データにハードコードしない方針（CLAUDE.md）。

追加する「3ベットレンジ」はユーザー提供のレンジ表画像から読み取ったもので、以下の26ハンドタイプ:

- ペア（4種 × 6 = 24）: AA, KK, QQ, JJ
- スーテッド（16種 × 4 = 64): AKs / A7s, A6s, A5s, A4s, A3s, A2s / K9s, K8s, K7s, K6s, K5s, K4s / QJs, QTs, Q9s
- オフスート（6種 × 12 = 72）: AKo, AQo, AJo / KQo, KJo / QJo

合計 24 + 64 + 72 = **160コンボ**。

## Goals / Non-Goals

**Goals:**
- 「3ベットレンジ」を3つ目のプリセットとして追加し、既存の出題フロー（初級プリセットレンジ問題・上級レンジ×ボード問題）で自動的に出題対象になること
- README・spec・テストを新レンジに合わせて整合させること

**Non-Goals:**
- レンジ選択ロジックの変更（等確率ランダムのまま）
- レンジ表 UI・問題文フォーマットの変更
- レンジのユーザー編集機能

## Decisions

- **id は `three-bet-range`、表示名は「3ベットレンジ」**: 既存の `open-range` / `call-range` の kebab-case 命名に合わせる。数字始まりの `3bet-range` は識別子として紛らわしいので避ける。
- **hands の記述順は既存2レンジと同じ「ペア → スーテッド（A→K→Q）→ オフスート」のグルーピング**: 定義の目視検算をしやすくするため。
- **合計160はテストの期待値としてのみ書く**: 定義データにはハードコードせず、`presetComboCount` の計算結果を検証する（既存方針の踏襲）。
- **`pickPresetRange` は無変更**: `PRESET_RANGES.length` ベースのランダム選択なので配列追加だけで3種の等確率選択になる。

## Risks / Trade-offs

- [レンジ表画像の読み取りミス] → design/spec にハンド集合を全列挙し、テストでハンドタイプ数（ペア4・スーテッド16・オフスート6）と合計160コンボを検証する。実装時に画像と突き合わせて確認する。
- [上級問題の負けコンボ計算が新レンジで想定外の値になる] → 展開・役評価は既存ロジックの再利用であり、既存テスト（デッドカード考慮）がカバーしている。
