# 設計: 上級4択の「総コンボ数に応じた分散数値」化

## Context

上級問題の4択は現在、固定境界 `ADVANCED_CHOICE_BUCKET_BOUNDS = [10, 40, 80]`（`src/config/gameConfig.ts`）から導出した4帯のラベル（「0〜10」「11〜40」「41〜80」「81以上」）。`questions.ts` の `bucketChoices()` が正解帯にフラグを立て、`makeAdvancedQuestion()` は正解帯の偏り防止のため目標帯を選んで再抽選している。

本来の意図は「自分のハンドが相手レンジに対してどのくらい強いか」の相対感覚を問うことなので、選択肢は相手レンジの総コンボ数（デッドカード除外後、`counts.total`）を基準に分散させた具体的な数値にする。例: total = 150 → 「3 / 27 / 67 / 129」。

制約（CLAUDE.md / 既存方針）:
- 難易度パラメータは `gameConfig.ts` に集約。マジックナンバーを散らさない
- 答え・選択肢はロジックで計算し、ハードコードしない
- テストは Vitest（導入済み）

## Goals / Non-Goals

**Goals:**
- 4択を「正解の負けコンボ数 + 総コンボ数の全域に分散した誤答3つ」の数値4択にする
- 誤答が正解の近傍値にならず、オーダーの違いが明確になる分散を保証する
- 正解が特定の位置・水準に偏らない（既存の偏り防止の意図を引き継ぐ）
- ゾーン分割の比率は設定ファイルで調整可能にする

**Non-Goals:**
- 初級・中級の誤答生成方式の変更
- 解説・リザルト画面の表示内容の変更（正確な負けコンボ数の表示は現状維持）
- 序盤ヒント（負けセル二重ハイライト）の変更
- 制限時間などの他の難易度パラメータの変更

## Decisions

### D1: ゾーン方式による誤答生成

総コンボ数 `total` を比率境界でスケールして4つの連続ゾーンに分割し、正解が属さない3ゾーンから誤答を1つずつ抽選する。

- 設定値: `ADVANCED_CHOICE_ZONE_RATIOS = [0.07, 0.27, 0.53]`（`gameConfig.ts`）
  - ゾーン境界 = `round(total × ratio)`。total = 150 なら境界 [11, 41, 80] → ゾーンは「0〜10 / 11〜40 / 41〜79 / 80〜150」相当
  - 初期比率は旧固定境界 10/40/80 を典型的な総コンボ数 150 に対する比率に換算したもの（プレイ感を見て調整する前提）
- 導出関数（`gameConfig.ts` に配置、テスト対象）:
  - `advancedChoiceZones(total): { min, max }[]` — 4ゾーンの整数区間を返す。隣接ゾーンは重複せず、0〜total の全整数がちょうど1つのゾーンに属する
  - `advancedZoneIndex(count, total): number` — 負けコンボ数が属するゾーンの index
- 誤答の抽選: 各ゾーンの中央寄り区間（`min + 0.2×span` 〜 `min + 0.8×span`、整数丸め）から一様乱数で1つ選ぶ。ゾーン端を避けることで、正解が境界近くに落ちた場合でも誤答との差が開きやすくなる

**代替案と却下理由:**
- 帯ラベルのまま境界だけ total 比例にする案 → ユーザーの例示は具体的な数値（3 / 27 / 67 / 129）であり、数値のほうが「実数を見積もる」訓練になる
- 正解×倍率（例: ×0.3 / ×3 / ×10）で誤答を作る案 → 正解が 0 や極小のとき破綻する。また選択肢が total の全域に散らばらず「レンジ全体に対する割合感覚」を問えない

### D2: 正解ゾーンの偏り防止は「候補ごとのゾーン」基準で継続

`makeAdvancedQuestion()` の再抽選ループは維持し、判定を `advancedBucketIndex(lose)`（固定帯）から `advancedZoneIndex(lose, total)`（候補の total から導出したゾーン）に置き換える。目標ゾーンを一様ランダムに選び、負けコンボ数がそこに落ちるまで再抽選（`ADVANCED_RETRY_LIMIT` で打ち切り、最終候補採用）。

これがないと lose は下位ゾーンに偏り、「いつも一番小さい選択肢を押せば当たる」クイズに退化する（既存と同じ理由）。

### D3: 選択肢は昇順ソート・正解フラグで判定

4値（正解1 + 誤答3）を昇順に並べて `Choice[]`（`label` = 数値文字列、`correct` フラグ）にする。シャッフルはしない。ゾーンが互いに素なので4値は必ず相異なり、正解の位置は正解ゾーンの位置に対応して問題ごとに変わる。正誤判定は既存どおり `correct` フラグ（ラベル比較に依存しない）。

旧方式の「位置固定による反射訓練」は、数値が毎問変わる新方式では成立しないため引き継がない。昇順は「小さい順に並ぶ」という視覚的な一貫性のために維持する。

### D4: 極小 total のガード

4ゾーンそれぞれに整数が1つ以上必要なため、`total` が小さすぎる候補（目安: `total < 20` を設定値 `ADVANCED_MIN_RANGE_TOTAL` として集約）は問題として成立しないものとし、再抽選ループ内で弾く。プリセットレンジは十分広く、リバー（デッド7枚）でも total は通常数十以上あるため、実運用でヒットすることはほぼない安全弁。

### D5: 旧バケット関連の削除

`ADVANCED_CHOICE_BUCKET_BOUNDS` / `ChoiceBucket` / `ADVANCED_CHOICE_BUCKETS` / `advancedBucketIndex` は参照がなくなるため削除する（後方互換を残す理由がない。localStorage 等の永続データには影響しない）。

## Risks / Trade-offs

- [正解がゾーン境界近くに落ちると隣ゾーンの誤答との差が縮む] → 誤答をゾーン中央寄り（20%〜80% 区間）から抽選して緩和。それでも差が総コンボ数の1割未満になるケースはありうるが、「大まかなオーダー把握」という目的上許容する
- [比率境界の初期値が体感に合わない可能性] → 設定ファイル1箇所（`ADVANCED_CHOICE_ZONE_RATIOS`）の変更で調整可能にしておく
- [再抽選条件が増える（目標ゾーン + 極小 total）ことで打ち切り率が上がる] → 極小 total は実プリセットではほぼ発生しないため影響は軽微。打ち切り時の最終候補採用は既存挙動を維持
- [乱数依存のテストのしにくさ] → ゾーン導出・ゾーン判定・選択肢の性質（4値相異・昇順・正解含有・各ゾーン1値）は決定的にテスト可能な形で関数を切る

## Migration Plan

1. `gameConfig.ts`: 新設定値・導出関数を追加し、旧バケット定義を削除
2. `questions.ts`: `bucketChoices` → 数値4択生成に置き換え、再抽選判定をゾーン基準に変更
3. テスト置き換え → `npm run build` / `npx vitest run` で確認
4. README の上級編・4択の記述を更新

ロールバックは git revert のみで可能（永続データ・外部依存なし）。
