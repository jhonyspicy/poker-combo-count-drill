## Context

現行はレベル進行型（Lv1 単純問題 → Lv2 レンジ表 → Lv3〜5 デッドカード）で、`gameConfig.ts` がレベル→タイプ→制限時間を決め、`PlayPage.tsx` がタイプごとに異なるレイアウトを描画している。自己ベストは単一記録（score/level/date）。

これを「初級/中級/上級の3難易度モード + 無限出題 + 連続正解数連動の時間短縮 + 難易度別自己ベスト」に再編し、問題画面を「レンジ表 / ボード / 文章」の3要素の出し分けに統一する。ビジュアルの正はデザインファイル `レンジ表とボード.dc.html`（Claude Design プロジェクト）で、その内容（配色・レイアウト・寸法）は本書 Decisions に反映済み。

制約: HashRouter 維持、localStorage のみ、ダーク固定、モバイルファースト、難易度パラメータは設定ファイル集約、答えは必ずロジック計算（CLAUDE.md ドメイン知識準拠）。

## Goals / Non-Goals

**Goals:**
- 難易度モード選択制（初級=レンジ表+文章 / 中級=ボード+文章 / 上級=レンジ表+ボード+文章）
- 連続正解数に応じた制限時間短縮（不正解 or 時間切れで即ゲームオーバー）
- 難易度ごとの自己ベスト記録
- プリセットレンジ（複数定義・都度生成しない）
- 上級用のハンド評価（勝敗判定）ロジック + テスト

**Non-Goals:**
- 旧自己ベストデータの移行（廃棄する）
- レンジ記法パーサ（"ATs+" 等）の実装 — プリセットはハンド表記の列挙で定義
- SNS共有・i18n（従来どおり将来候補）
- 文章のみの問題タイプ（ユーザー補足で「今は思いついていない」— データモデル上は3要素とも省略可能にして拡張余地だけ残す）

## Decisions

### D1. 難易度はルートパラメータで表現する
`/play/:difficulty`（`beginner` | `intermediate` | `advanced`）とする。navigation state と違いリロードや直リンクでも壊れず、リザルトからの「もう一度」も同一URLへの遷移で済む。不正な difficulty はトップへリダイレクト。ResultPage への受け渡しは現行どおり navigation state（score・question・difficulty・isNewBest）。

### D2. 設定は gameConfig.ts に難易度別テーブルとして集約する
```ts
type Difficulty = 'beginner' | 'intermediate' | 'advanced';
DIFFICULTY_CONFIG: Record<Difficulty, {
  label: string;              // 初級編 / 中級編 / 上級編
  baseTimeSec: number;        // 基準時間（初期値: 初級12 / 中級30 / 上級45）
  streetTimeBonusSec: { flop: 0; turn: number; river: number }; // ボードあり問題の街別加算（中級・上級）
  timeScaleFactor: number;    // 正解ごとの短縮係数（初期値 0.95）
  timeFloorRatio: number;     // 下限（初期値 0.5）
}>
```
制限時間 = `(baseTimeSec + 街別加算) × timeScaleFactor^streak`、下限 `baseTimeSec × timeFloorRatio`。旧 `getLevelType` / `MASTERY_STREAK` / レベル連動スケーリングは削除。数値はすべてプレイ感で調整する前提の初期値。

### D3. Question 型は「3要素の出し分け」を optional フィールドで表現する
既存の `Question` を踏襲し、`rangeCells?`（レンジ表）・`board?`/`hero?`（ボード）・`text`（文章、必須）の有無で描画を出し分ける。判別 union にはしない — PlayPage は「あるものを描く」だけでよく、3要素とも省略可能な将来の文章のみ問題にもそのまま対応できるため。`type` は `QuestionType = 'beginner-range' | 'board-count' | 'range-vs-board'` に再編し、レンジパネルの見出し（「レンジ表」/「相手のレンジ」）を切り替えるための `rangeLabel?` を追加する。

### D4. 初級問題は rangeQuestions のパターン方式を拡張して作る
既存 `RangePattern`（test 関数 + ラベル）に単一ハンド系パターンを追加する:
- 単一セル: 「AKsは何コンボ？」「AKoは何コンボ？」「AAは何コンボ？」（ランクはランダムに選ぶ。答えは 4 / 12 / 6 だがロジックで算出）
- ハンドグループ: 「AK（s+o合計）」「Ax（Aを含む全ハンド）」既存の suited / offsuit / pairs / 全体(1326) など
- プリセットレンジ: 「相手のレンジ表の通りのとき、相手が持ちうるのは何コンボ？」— プリセットレンジのセル合計（ペア6/スーテッド4/オフスート12）

いずれも該当エリアをハイライトした 13×13 グリッドを表示する。答えは `cellCombos` の合計で計算（ハードコードしない）。旧 SIMPLE_POOL（文章のみ・固定キュー・全問正解制）は廃止し、初級はランダム無限出題に統一する。

### D5. 中級問題は現行デッドカード問題をそのまま継承する
`makePairQuestion` / `makeSuitedQuestion` / `makeOffsuitQuestion` と comboCalculator は変更なし。フロップ/ターン/リバーは 1/3 ずつランダム選択（旧レベル固定を廃止）。解説の AK − AKs = AKo 導出順も維持。

### D6. ハンド評価は「C(n,5) 全列挙 + 数値スコア」で実装する
新規 `src/lib/handEvaluator.ts`:
- `evaluate5(cards: Card[]): number` — 5枚の役を単一数値にエンコード（カテゴリ × 15^6 + タイブレーク5枚を15進で連結）。ホイール（A-5）ストレートも処理
- `evaluateBest(cards: Card[]): number` — 5〜7枚から C(n,5) ≤ 21 通りを全列挙して最大値
- `compareHands` 相当は数値比較（大きい方が強い、同値はチョップ）

上級の相手コンボは最大 ~150 × 21評価 = 高々数千回の evaluate5 で、リトライ込みでも実用上一瞬。ルックアップテーブル等の最適化は不要。**正解計算の根幹なので Vitest を導入し、役カテゴリ・タイブレーク・ホイール・ボードプレイ（チョップ）のテストを必須とする**（既存 `__tests__` の体裁に合わせる）。

### D7. 上級問題は「負け」「勝ち」両方を出題し、リトライで答えを可算域に収める
生成手順: プリセットレンジをランダム選択 → ボード（フロップ/ターン/リバー）+ 自分ハンド2枚を配る → レンジ内の各セルをデッドカード除外で具体コンボに展開 → 各コンボと自分の役を比較。
- 「何コンボに**負けて**いますか？」= 自分より強いコンボ数（チョップは含めない）
- 「何コンボに**勝って**いますか？」= 自分より弱いコンボ数（チョップは含めない）
- 答えが 1〜60 の範囲に入るまで再抽選（上限40回、超えたら最後の候補を採用）。60 超は人間が制限時間内に数える問題として成立しないため

解説はリザルト画面用に「相手のレンジ ○○ コンボ中、あなたの役（例: ワンペア・K）より強いのは △△ コンボ」の形式で組み立てる。

### D8. プリセットレンジはハンド表記の列挙で定義する
新規 `src/lib/presetRanges.ts` に `{ id, name, hands: string[] }`（hands は "AA" / "AKs" / "AKo" 形式の169表記）で3〜5種類定義（例: タイトオープン / ブロードウェイ中心 / ルースディフェンス）。デザインファイルの inRange（55+ / Axs / ブロードウェイ / スーテッドコネクター）を1種類目のベースにする。表記→グリッド座標の展開と合計コンボ数はロジックで計算し、定義の妥当性（表記の正当性・重複なし）を Vitest で検証する。都度生成しないためユーザーは「このレンジ」を覚えて練習できる。

### D9. プレイ画面レイアウトはデザインファイル準拠の縦積み構成
上から: ヘッダー（難易度名 + 連続正解数）→ タイマーバー → [レンジ表パネル] → [フェルト帯（ボード | 仕切り | あなた）] → 問題文 → フィードバック行（高さ固定28px）→ 2×2 選択肢グリッド。
- レンジ表パネル: max-width 330px、`#111a2e` 背景 + `#263252` 枠、見出し（初級=「レンジ表」/ 上級=「相手のレンジ」）と凡例チップ、13×13 グリッド（gap 2px・aspect-ratio 1/1・ハイライト `#f5b83d`）
- フェルト帯: 現行の楕円テーブルをやめ、デザインファイルの**角丸帯**（radial-gradient フェルト + 木枠 border）に変更。ボード（42×60px）と自分のハンド（44×62px）を縦仕切り線で並べる
- 選択肢・フィードバック・タイマーバーの配色は現行 PlayPage の値（デザインファイルと同一系統）を踏襲
- 3要素同時表示（上級）でも 430px 幅 × 縦 880px 程度に収まることをモバイルビューで確認する
既存 `RangeGrid` / `PlayingCard` コンポーネントはサイズ・見出しを props 化して再利用する。

### D10. 自己ベストは新キーに難易度別で保存し、旧データは廃棄する
`poker-drill-best-v2` に `Partial<Record<Difficulty, { score: number; date: string }>>` を保存。旧キー `poker-drill-best` は読み込まず、初回保存時に削除する（level 概念が消えるため移行は無意味）。トップページは3難易度ボタンそれぞれに自己ベストを添えて表示する。

### D11. デバッグ用レベル開始は削除する
難易度選択がトップページの標準機能になるため、`debugStartLevel` まわりのコード・スペックごと削除する。

## Risks / Trade-offs

- [上級の答えが実戦感覚とずれる（チョップ扱い・ボードプレイ等）] → チョップは勝ち負けに含めないと問題文・解説・スペックで明示。handEvaluator のテストでボードプレイ（全員同役）ケースを固定
- [上級のリトライ生成が収束しない（レンジとボードの組み合わせ次第で答えが常に域外）] → 上限40回で打ち切り、最終候補をそのまま採用（answer≥1 は保証）。プリセットレンジは合計100〜200コンボ程度に設計して収束しやすくする
- [初級の単一セル問題は答えが 6/4/12 に固まり4択が単調になる] → 誤答は近傍値ではなく典型的な取り違え（4↔6↔12↔16）を優先して生成する
- [自己ベストのリセットにユーザーが気づかず混乱] → README とリザルト/トップの表示が新形式に完全に切り替わるため実害は小さい。個人開発アプリでありユーザー告知は不要と判断
- [時間短縮パラメータの初期値が厳しすぎ/緩すぎ] → すべて gameConfig の1テーブルに集約済みで、プレイ感を見て数値のみ調整できる
