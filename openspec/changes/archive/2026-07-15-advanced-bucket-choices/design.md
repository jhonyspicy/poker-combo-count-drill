# Design: 上級問題のバケット4択

## Context

上級問題（range-vs-board）の4択は `generateAdvancedDistractors`（questions.ts）が正解の ±1〜4 ステップ（step は答えの大きさで 1 / 2 / 4）で誤答を作っており、選択肢同士が最大でも ±16 しか離れない。正確なカウントを前提にした設計で、実戦の「負けている可能性のオーダーを瞬時に見積もる」スキルとずれている。

また、答えを 1〜60 に収めるための再抽選（`ADVANCED_ANSWER_MIN/MAX`）は「人間が制限時間内に正確に数えられる問題にする」ための制約であり、バケット4択ではこの前提自体が不要になる。

選択肢は `Question.choices: number[]` として保持され、PlayPage が `choice === question.answer` で正誤判定している。バケットは数値ではなくラベル（「11〜40」など）なので、choices の型を一般化する必要がある。

## Goals / Non-Goals

**Goals:**
- 上級の4択を大きく離れたコンボ数帯にし、オーダー感覚の訓練に最適化する
- バケット境界を設定ファイル（gameConfig.ts）に集約し、チューニング可能にする
- 正解バケットが偏らないようにする（4つのボタンすべてが「生きた」選択肢であり続ける）

**Non-Goals:**
- 初級・中級の選択肢生成の変更
- 制限時間（baseTimeSec 45秒）の変更。バケット化で見積もりが速くなるはずだが、時間チューニングはプレイ感を見て別途行う
- 序盤ヒント（負けセル二重ハイライト）・解説文・リザルト画面の変更

## Decisions

### D1: バケットは固定4帯・境界は設定値・上端はオープンエンド

`gameConfig.ts` に境界値 `ADVANCED_CHOICE_BUCKET_BOUNDS = [10, 40, 80]` を置き、4帯 `[0..10] [11..40] [41..80] [81..∞)` を導出する。ラベル（「0〜10」「11〜40」「41〜80」「81以上」）とバケット判定関数も境界値から導出し、ハードコードしない。

- ユーザー案の 0〜10 / 10〜40 / 40〜80 / 80〜150 をベースに、境界の重複を解消（答えが 10 のとき正解が2つになる曖昧さを排除。区切りのいい上限値 10 / 40 / 80 は表示に残す）
- 最上位帯を「81以上」にする理由: レンジの総コンボ数はプリセット（オープン/コール/3ベット）とデッドカードで毎回変わるため、150 のような固定上限は問題によって嘘になる。オープンエンドなら常に正しい
- 代替案（不採用）: 総コンボ数比例の相対バケット → 表示数字が毎回変わり帯の感覚が身につかない。割合ベース → コンボを数えるというアプリの主旨から離れる

### D2: 選択肢は昇順固定・シャッフルしない

上級の4択は常に「0〜10 / 11〜40 / 41〜80 / 81以上」の昇順でボタンに並べる。位置が固定されることで「危険度メーター」のような反射的判断の訓練になる。シャッフルによる当てずっぽう防止は不要（4帯すべてが常に表示されるため、位置を覚えても答えは分からない）。

### D3: Question.choices を「ラベル + 正解フラグ」に一般化

```ts
export interface Choice {
  label: string;   // ボタン表示（初級・中級は数値の文字列、上級はバケット帯）
  correct: boolean; // 正解か
}
```

- `Question.choices: Choice[]` に変更。`Question.answer: number` は正確な負けコンボ数のまま残す（解説・リザルト画面用）
- PlayPage の正誤判定は `choice === question.answer` から `choice.correct` に変更。ボタンの key は index にする（バケットラベルは問題間で同一のため）
- 代替案（不採用）: `choices: number[]` を維持し上級だけ `choiceLabels?: string[]` を追加 → 判定とラベルの対応が暗黙になり事故りやすい。判定情報を choices 自体に持たせる方が単純

### D4: 目標バケット狙いの再抽選で正解バケットを均等化

`makeAdvancedQuestion` の再抽選を「答えが 1〜60 に入るまで」から「目標バケットに入るまで」に差し替える:

1. 問題生成の冒頭で目標バケットを一様ランダムに選ぶ
2. 候補（レンジ・ボード・ハンド）を抽選し、負けコンボ数が目標バケットに落ちたら採用
3. `ADVANCED_RETRY_LIMIT`（40回、既存値を流用）で打ち切り、最後の候補を採用

打ち切り時のフォールバックが単純になる点に注意: バケット4択では全候補が有効な問題（どのバケットに落ちても正解の帯が必ず存在する）なので、現行のような「lose >= 1 の候補を優先保持する」補正は不要。`ADVANCED_ANSWER_MIN / ADVANCED_ANSWER_MAX` は削除する。

- 均等化しない案（不採用): 抽選の自然な分布は低い帯に偏ると予想され、「いつも同じボタンで当たる」クイズに退化する
- 「81以上」「0〜10（特に0）」が目標のときは再抽選回数が増えうるが、1候補の評価コストは現行と同じ（レンジ全コンボのハンド評価）で、上限も現行と同じ40回のため、体感悪化はない見込み

### D5: 誤答生成ロジックから上級を切り離す

`generateAdvancedDistractors` と `generateSteppedDistractors` の上級用途を削除する（`generateSteppedDistractors` は初級の `generateRangeDistractors` が使い続けるので関数自体は残す）。上級の choices は誤答生成ではなく「4帯の列挙 + 正解帯のフラグ立て」で作る。

## Risks / Trade-offs

- [答えが 0 の問題が出題可能になる] → 「あなたは何コンボに負けていますか？」に対し正解「0〜10」は自然に成立する。解説文は現行のまま `強い 0 / 弱い N` を表示でき、破綻しない。ヒント表示も負けセルが無いだけで正常動作する
- [負けコンボ数が大きい問題（60超）はレンジ表の情報量が多く、見た目が重くなる] → バケット判定に必要なのは正確なカウントではなく順序見積もりなので、制限時間内に判断可能。プレイ感が悪ければ境界値や目標バケットの重みを設定ファイルで調整する
- [既存の自己ベストとの連続性] → スコアは連続正解数のままで互換。難易度は下がる方向（4択の分解能が粗くなる）なので、既存ベストが不当に不利になることはない
