## Context

モバイルブラウザの `100vh` は「アドレスバーが隠れた状態」の高さで固定されるため、`min-h-screen` を使う画面はアドレスバー表示中にその高さ分だけはみ出し、コンテンツが収まっていても縦スクロールが発生していた。プレイ画面（`PlayPage.tsx`）のみ既に `min-h-dvh` を採用済みで、トップ・遊び方・リザルトの3画面が `min-h-screen` のまま残っていた。

本変更は実装済みであり、この design は採用した判断の記録である。

## Goals / Non-Goals

**Goals:**
- コンテンツがビューポートに収まる画面で、アドレスバー分の余分なスクロールをなくす
- 全画面で最小高さの基準を `min-h-dvh` に統一する

**Non-Goals:**
- スクロール自体の禁止（遊び方画面などコンテンツが長い画面は引き続きスクロール可能）
- `overflow: hidden` や `position: fixed` による強制的なスクロールロック
- アドレスバーの出没に伴うレイアウトアニメーションの制御

## Decisions

- **`dvh`（dynamic viewport height）を採用**: `svh`（small = アドレスバー表示時の高さ）だとアドレスバーが隠れたときに下部が余り、`lvh`（large）だと `vh` と同じ問題が残る。`dvh` は実際に見えている高さに動的に追従するため、`min-h-` 用途に最適。プレイ画面が既に `min-h-dvh` で問題なく動作している実績もある。
- **Tailwind 標準ユーティリティ `min-h-dvh` を使用**: Tailwind CSS v4 に組み込みで存在するためカスタム CSS 不要。
- **`min-h` を維持（`h-dvh` にしない）**: 固定高さにするとコンテンツが長い画面（遊び方）で内容が切れる。最小高さのままにして自然なスクロールを許容する。
- **`dvh` 非対応の古いブラウザへのフォールバックは設けない**: 対象は現行のモバイルブラウザ（iOS Safari 15.4+ / Chrome 108+ で対応済み）であり、仮に非対応でも `min-h` が効かず縦に縮むだけで致命的な破綻はない。

## Risks / Trade-offs

- [アドレスバー出没時に `dvh` が変化し、画面全体の高さが動く] → `min-h` 用途では中身の配置が伸縮するだけで、実用上の違和感は小さい。固定要素には使っていない。
- [ゲームオーバー時など画面下端基準のUIがアドレスバー出没で微妙に動く] → 現状のUIは `mt-auto` などフロー配置のみで、固定配置に依存していないため影響は軽微。
